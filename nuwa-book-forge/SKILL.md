# 女娲 · 炼书术

> 「把混沌的信息锻造成有序的知识——这是炼书，不是搬运。」

## 核心理念

女娲炼书不是搜索结果的搬运工，是**知识结构的锻造者**。

一本好的技术调研书具备：
- **结构清晰**：从「是什么」到「怎么做」到「走向哪」的完整认知路径
- **有据可查**：每个关键论点都有可验证的来源支撑
- **实用导向**：不是论文综述，是帮读者真正理解并能行动的指南
- **时效性强**：包含最新的发展和趋势，不是过时的资料汇编

**关键区分**：提炼的是 WHY 和 HOW，不是堆砌 WHAT。

## 执行流程

### Phase 0: 入口与需求澄清

收到用户输入后，确认以下信息：

| 确认项 | 说明 | 默认值（用户没说时） |
|--------|------|-------------------|
| **主题** | 调研什么技术/概念？边界在哪？ | 用户输入的原话 |
| **目标读者** | 技术人员？管理者？初学者？ | 技术人员 |
| **深度与篇幅** | 快速概览(~5000字,3-5章) / 标准调研(~15000字,8-12章) / 深度专著(~30000字,15+章) | 标准调研 |
| **本地语料** | 有没有相关论文、文档、代码仓库？ | 无，走网络搜索 |
| **输出语言** | 中文为主 / 英文为主 / 双语 | 中文为主 |

用户说「就做XX」没有更多信息 → 直接用默认值推进，不追问。

### Phase 1: 多维信息采集（并行Agent Swarm）

#### 模式判断

| 模式 | 触发条件 | 策略 |
|------|---------|------|
| **纯网络搜索**（默认） | 用户没有提供本地素材 | 6个Agent全部走网络搜索 |
| **本地语料优先** | 用户提供了论文/文档/代码等 | 先分析本地素材，网络搜索补充 |
| **纯本地语料** | 用户明确说「只用我给的素材」 | 只分析本地素材，不做网络搜索 |

**本地语料优先模式的执行逻辑**：
1. 先读本地素材，按6个维度分类
2. 识别信息缺口，只对缺失维度启动网络搜索Agent
3. 调研文件中明确区分「来自用户提供素材」vs「来自网络搜索」

#### 6个Agent的任务分配

启动6个并行subagent，每个负责不同信息维度。

| Agent | 搜索目标 | 提取重点 | 输出文件 |
|-------|---------|---------|---------|
| 1 基础 | 论文、教材、官方文档、RFC/标准 | 核心定义、关键术语表（中英对照）、发展时间线、奠基性论文、核心问题域 | `01-foundations.md` |
| 2 架构 | 技术博客、源码分析���架构文档、论文 | 系统架构图描述（Mermaid语法）、核心算法、数据流、关键设计决策、实现原理 | `02-architecture.md` |
| 3 生态 | GitHub、包管理器、框架官网、benchmark | 主流工具/库对比表、选型指南、社区活跃度、兼容性、版本历史 | `03-ecosystem.md` |
| 4 应用 | 案例研究、生产实践、技术分享、开源项目 | 真实部署案例、性能数据、踩坑记录、最佳实践、典型使用场景 | `04-applications.md` |
| 5 挑战 | 技术批评、已知issue、对比分析、Reddit/HN讨论 | 技术局限、常见误解、竞品对比、失败案例、适用边界 | `05-challenges.md` |
| 6 趋势 | 最新论文(arXiv)、行业报告、roadmap、KOL观点 | 近6个月动态、路线图、预测、新兴替代方案、学术界/工业界分歧 | `06-trends.md` |

#### 每个Agent的硬性要求
- 调研结果必须写入对应 `references/research/0X-xxx.md`
- 注明信息来源URL和可信度
- 区分「一手」（官方文档/论文/源码/标准）vs「二手」（博客/教程/评论）
- 技术术语中英对照，首次出现给出英文原文
- 发现矛盾或争议时保留，不要只呈现一面

#### Agent prompt模板

spawn subagent时，用以下结构给任务（以Agent 1基础为例）：

```
你的任务：���度调研[主题]的基础知识。

搜索方向：
- 核心定义：这个技术/概念是什么？解决什么问题？
- 关键术语表：中英文对照，每个术语一句话解释
- 发展时间线：从诞生到现在的重要里程碑
- 奠基性论文/文档：最早提出这个概念的重要文献
- 核心问题域：这个领域关注的核心问题有哪些？

输出要求：
- 写入 [工作目录]/references/research/01-foundations.md
- 每条信息标注来源URL
- 区分一手（官方/论文）vs 二手（博客/教程）
- 术语首次出现标注英文原文
- 如果有不同定义/理解，并列呈现
```

其他5个Agent按同样结构调整搜索方向和输出文件名。

#### 信息源优先级

| 来源类型 | 揭示什么 | 权重 |
|---------|---------|------|
| 官方文档/规范 | 权威��义、设计意图 | 最高 |
| 学术论文 | 理论基础、实验数据 | 最高 |
| 源码/开源项目 | 真实实现、设计决策 | 最高 |
| 技术博客（作者原创） | 实践经验、深入分析 | 高 |
| 生产案例 | 真实效果、踩坑记录 | 高 |
| 社区讨论 | 常见问题、争议点 | 中等 |
| 教程/入门文章 | 入门视角、常见误解 | 中等 |
| 二手转述/新闻 | 趋势信号 | 低 |

#### 信息源黑名单（永远排除）

- **知乎**：洗稿严重、信息失真率高
- **微信公众号**：封闭生态、无法验证
- **百度百科/百度知道**：信息陈旧且不可靠

中文渠道只接受：36氪、极客公园、晚点LatePost、少数派、机器之心、InfoQ中文、掘金（原文）等。

#### Agent超时与失败处理

- **单个Agent超时**（搜索5分钟无有价值结果）：不等待，继续推进。在Phase 2中标注「信息不足」
- **信息源匮乏**（某Agent来源<5条）：Phase 1.5提醒用户，相关章节降低期望
- **Agent结果冲突**：保留矛盾——在书中呈现为「不同观点的对比」，这是有价值的内容

### Phase 2: 书籍结构设计

根据调研结果，设计书籍大纲。

#### Step 1: 读取模板

读取 `references/book-template.md` 获取书籍结构规范。

#### Step 2: 分析调研结果，规划章节

基于6个Agent的发现，确定：
- 哪些发现足以独立成章？
- 哪些发现应该合并？
- 信息的逻辑流是什么？（概念→原理→实践→展望）

#### Step 3: 生成大纲

标准四部分结构（根据深度调整章节数）：

**快速概览模式**（3-5章）：
```
第1章：概述与核心概念（Agent 1）
第2章：工作原理（Agent 2）
第3章：实践与应用（Agent 3+4）
第4章：局限与展望（Agent 5+6）
附录：术语表 + 参考资料
```

**标准调研模式**（8-12章）：
```
前言

第一部分：基础篇
  第1章：概述与动机（Agent 1）
  第2章：核心概念详解（Agent 1）
  第3章：发展历程（Agent 1 + 6）

第二部分：原理篇
  第4章：技术架构（Agent 2）
  第5章：核心机制详解（Agent 2）
  第6章：关键技术与权衡（Agent 2 + 5）

第三部分：实践篇
  第7章：生态与工具全景（Agent 3）
  第8章：选型与实践指南（Agent 3 + 4）
  第9章：典型案例分析（Agent 4）

第四部分：展望篇
  第10章：挑战与局限（Agent 5）
  第11章：未来趋势（Agent 6）

附录
  A：术语表
  B：参考资料
  C：推荐阅读
```

**深度专著模式**（15+章）：
在标准模式基础上，每个部分拆分更多章节，增加「进阶专题」「深度源码分析」「行业应用矩阵」等章节。根据调研结果的具体丰富度灵活调整。

#### Step 4: 展示大纲给用户确认

大纲中标注每个章节的主要数据来源（哪个Agent），让用户了解信息基础。

用户确认OK → Phase 3。
用户想调整结构 → 修改后再确认。

### Phase 3: 内容撰写

按确认的大纲逐章撰写。

#### 每章结构

```markdown
# 第N章：[章节标题]

> [一句话点明本章核心价值]

## [节标题]

[正文内容，含代码示例/图表/表格]

---

**本章要点**
1. [要点1]
2. [要点2]
3. [要点3]

**延伸阅读**
- [相关章节交叉引用]
- [外部资源链接]
```

#### 撰写规则

1. **术语规范**：技术术语首次出现时给出中英文定义，格式：`术语（English Term）：定义`
2. **代码示例**：用围栏代码块并标注语言，可运行的代码不要标注伪代码，确实无法运行的标注为「伪代码」
3. **图表**：
   - 架构图用 Mermaid 语法（方便PDF渲染）
   - 对比用 Markdown 表格
   - 流程用 Mermaid flowchart
4. **来源标注**：关键论点用 `[^1]` 脚注标注来源，章末或文末汇总参考文献
5. **观点平衡**：有争议的话题呈现多方观点，不偏袒一面
6. **交叉引用**：相关章节之间用「详见第X章」互相链接

#### 前言撰��

前言包含：
- **为什么写这本书**：这个技术/概念为什么值得深入了解
- **本书结构**：各部分的内容概览和阅读路径建议
- **读者指南**：需要什么前置知识、不同读者如何选读
- **调研说明**：信息来源、调研时间、局限性

#### 附录撰写

- **术语表**：全书术语的中英对照表，按拼音/字母排序
- **参考资料**：按章节分组，每个来源标注类型（论文/官方文档/博客等）
- **推荐阅读**：精选5-10个最值得深入的资源，附推荐理由

## 品味守则

| 原则 | 一句话 |
|------|--------|
| 结构 > 堆砌 | 一本结构清晰5000字的书 > 一本没有结构的50000字资料汇编 |
| 实用 > 全面 | 帮读者真正理解并能行动，比面面俱到更重要 |
| 来源 > 观点 | 有据可查的论点 > 看起来专业但无法验证的说法 |
| 新 > 旧 | 2026年的信息比2020年的更有价值（除非是奠基性内容） |
| 深入 > 广泛 | 一个概念讲透比十个概念浅尝辄止更好 |

### 绝不做的事
- 编造不存在的论文/数据/案例
- 把搜索结果的摘要直接拼凑成章节
- 忽略争议和局限只呈现优点
- 在信息不足时强行写满字数
- 省略来源标注让内容看起来像是「权威论述」

## 特殊场景

### 非技术主题

当用户调研的不是技术而是其他领域（如「冥想的科学依据」「断食与健康」）：
- Agent 2（架构）调整为「机制/原理」：搜索科学研究、实验数据
- Agent 3（生态）调整为「方法/流派」：搜索不同学派、实践方法
- 其他Agent保持不变
- 来源优先级调整为：学术论文 > 权威机构报告 > 科普文章 > 社区经验

### 超热门话题（信息爆炸）

当调研对象是极度热门的话题（如"大语言模型"、"AI"）：
- Phase 0 就提醒用户：「这个话题信息量极大，建议聚焦某个子方向」
- 如果用户坚持全面调研 → 切换到深度专著模式
- Agent 搜索时增加时间过滤（优先近1年内容）
- 书籍结构增加「快速导航」章节，帮读者按兴趣选读

### 冷门话题（信息稀少）

当 Phase 1.5 发现总来源 < 15 条：
- 提醒用户：「这个主题公开资料较少，书籍深度会受限」
- 切换到快速概览模式（3-5章）
- 在前言中明确标注信息局限性
- 建议用户提供本地资料补充

## 最后

炼书不是搬运信息，是锻造认知。

一本好的调研书，让读者从「听说过这个名字」变成「真正理解并能做出判断」。不是替代读者去思考，而是给他们思考所需的完整素材和清晰框架。
